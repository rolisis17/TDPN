Onboarding steps

Sign in, then choose your lane

Start with wallet sign-in, then continue in the client lane, operator lane, or both based on your role.

Step 1

Wallet sign-in

Choose a wallet, request a challenge, then verify with your signature to create or refresh the session.

Step 2

Client registration

Use the signed-in session to register the client, set the path profile, and optionally provide bootstrap details.

Step 3

Operator workflow

Apply the operator role, check readiness, and approve the server workflow once the application is eligible.

Step 1

Wallet sign-in

Start here to generate a challenge, capture the wallet signature, and establish a session both lanes can reuse.

Keplr and Leap wallet-extension sign + verify are available now.

Manual verify is available in compatibility mode.

Step 2

Client registration

Register the signed-in client against the selected path profile. Bootstrap directory and invite values stay optional for compatibility.

Step 2 readiness
Client status: Not signed in Sign in first, then register the client profile.

The client lane stays visible for every role. If your role is server-only, this banner explains the lock reason and next path.

Compatibility override is off by default. Keep it off for standard session-based registration.

Step 3

Operator workflow

This is the server lane. Sign in first, then apply the operator role, check readiness, and approve only when the session is eligible.

Operator readiness
Application status: Not signed in Sign in first to unlock operator onboarding.
Endpoint trust posture
Endpoint posture: Unknown Refresh server status to load endpoint posture and warnings.

Use the readiness banner to confirm whether the current session can proceed as an operator or needs another sign-in or approval cycle.

Approval actions use session token auth by default; admin token is a compatibility fallback. Rejection always requires a moderation reason.
