Onboarding lanes

Choose the path that matches the role.

Both paths start with wallet sign-in. After that, clients proceed immediately while operator access remains intentionally gated by application review and approval.

Client self-serve lane

Register and connect without waiting for manual approval.

This lane is designed for trusted clients that need to join quickly with policy-aware defaults and clear registration feedback.

  1. Sign in with wallet: Create or refresh an authenticated session.
  2. Register client profile: Choose path profile and submit registration.
  3. Operate as client: Continue with role-appropriate actions immediately.
Open client registration in portal

Operator approval lane

Apply for operator access, then continue after admin approval.

Operator privileges are permissioned by design so elevated actions only unlock after an explicit application and approval decision.

  1. Sign in and apply: Submit operator ID and application details.
  2. Undergo review: Admins verify readiness and moderation context.
  3. Run operator actions: Approval unlocks server lifecycle controls.

Flow cues

What users should expect in each onboarding experience.

Client self-serve cues

  • Immediate progression from wallet sign-in to client registration.
  • Readiness guidance focuses on session and registration completeness.
  • No operator approval dependency for client-only operations.

Operator approval cues

  • Application status remains visible until reviewed and approved.
  • Readiness messaging explains lock reasons when access is not yet eligible.
  • Operator lifecycle actions unlock only after successful approval state.
Why GPM

Product-grade controls from first login to steady operations.

Trusted access posture

GPM starts from a pinned trusted domain, then extends into signed directories with local fallback so teams keep a predictable trust model under changing conditions.

Mesh transport with graceful fallback

Direct mesh connectivity stays preferred while managed relay routing preserves continuity when peers are unreachable.

Role-aware operations

Client onboarding is self-serve, operator activation is permissioned, and each transition is visible to the people responsible for review.

Wallet-native sign in

Sessions are wallet-authenticated by default, keeping identity handoff clear without credential sprawl.

Clear cross-surface language

Desktop and web use shared onboarding language so operators, support teams, and clients stay aligned on status and next actions.

Operator-grade visibility

Application and approval states are explicit, reviewable, and designed to avoid hidden background transitions.

Trust and governance

Controls for teams that need confidence, not just connectivity.

Approval trails

Operator applications and lifecycle changes are designed to be reviewable so teams can trace who approved what and when.

Least-privilege role separation

Client and operator workflows remain separate so elevated access is granted only after the right preconditions are satisfied.

Secure bootstrap strategy

Initial trust is explicit, while downstream discovery uses signed, durable directories and controlled fallback behavior.

Operational clarity

Status messages and workflow states stay consistent so teams can act quickly with fewer assumptions.
